2. نقطة مهمة ظهرت أثناء المحطة — مبدأ "تفضيل المجاني"

أثناء قرار النماذج (7)، عبّر المستخدم عن رغبته في موقع مجاني بالكامل بدل Fluent Forms Pro المدفوعة الموصى بها في الخطة. متفترضتش إن ده يخص النماذج بس — رجعت سألت توضيح صريح هل التفضيل عام على كل القرارات المتبقية ولا خاص بالنماذج، فأكّد المستخدم إنه عام على كل القرارات المتبقية. على أساس كده اتغيّر قرار 8 (SureCart) لباقة Launch المجانية بدل Pro، وقرار 11 (ShortPixel) للباقة المجانية. القرارات اللي اتعتمدت قبل ما يُذكر هذا التفضيل (Bricks Builder وACF Pro، قرارات 3 و5، وهما مدفوعين) فضلوا زي ما هم بدون مراجعة — لم يُطلب التراجع عنهم صراحةً.

هذا المبدأ ("عند وجود بديل مجاني عملي، الأفضلية له ما لم يُذكر غير ذلك") اتسجل كملاحظة عامة دائمة في APPROVED-DECISIONS.md.

المحطة 3 — تركيب ثيم shemo-child الفرعي

التاريخ: 2026-07-01 · ✅ مكتملة · النوع: تعديل فعلي على موقع LocalWP — تثبيت ثيم/إضافة + تنفيذ Git tracking لأول مرة

السياق

المحطة دي مخصصة لتنفيذ القرار رقم 4 المعتمد (ثيم فرعي مخصص shemo-child)، اعتمادًا على القرار رقم 3 (page builder). قبل أي خطوة تقنية، اتسأل المستخدم صراحةً: هل ترخيص Bricks Builder متاح بالفعل ولا لسه محتاج شراء؟ — بما إن Bricks مالوش نسخة مجانية إطلاقًا (ترخيص one-time من $79). ردّ المستخدم إنه مش هيشتري حاجة، عاوز كل حاجة مجانية.

1. مراجعة وتعديل القرارات المعتمدة (قبل أي تنفيذ)

ده تعارض مباشر مع القرار رقم 3 المُقفَل فعليًا في المحطة 2. اتسأل المستخدم تحديدًا: يرجع عن Bricks لبديل مجاني، يوقف المحطة، ولا يشتري لاحقًا بس يستنى؟ اختار المستخدم صراحةً: الرجوع عن Bricks → GeneratePress (مجاني)، وهو البديل (Runner-up) الموثّق أصلاً في MASTER-PLAN.md §8.

APPROVED-DECISIONS.md اتعدّل قبل أي تنفيذ تقني:

  • القرار 3 الأصلي (Bricks): بقى ⛔ Superseded، مع توثيق سبب وتاريخ التراجع، بدون حذف السجل التاريخي.
  • قرار 3 (مُعدَّل) جديد: ✅ GeneratePress (مجاني) + GenerateBlocks (مجاني) — معتمد.
  • القرار 4 (الثيم): اتعدّل ليصبح shemo-child فوق GeneratePress بدل Bricks (استراتيجية الثيم الفرعي نفسها فضلت زي ما هي).
  • ملاحظة "تفضيل المجاني" العامة: اتحدّثت لتوثّق إن القرار 5 (ACF Pro، لسه مدفوع ولم يُراجَع) لازم يتعرض على المستخدم صراحةً في بداية محطة shemo-core/نمذجة المحتوى قبل أي شراء.

2. إعداد بيئة التنفيذ (wp-cli)

اتعمل اكتشاف جديد للمسارات الفعلية على الجهاز (متسجلتش بالتفصيل من محطة 1):

  • wp-cli الثنائي: C:\Program Files (x86)\Local\resources\extraResources\bin\wp-cli\win32\wp.bat (مُجمَّع جوه تطبيق Local نفسه)
  • PHP 8.3.29: ...\Local\lightning-services\php-8.3.29+1\bin\win64
  • MySQL client 8.4.0: ...\Local\lightning-services\mysql-8.4.0\bin\win64\bin
  • PHPRC: ...\Local\run\<site-run-hash>\conf\php
  • MYSQL_HOME: ...\Local\run\<site-run-hash>\conf\mysql (port 10004، root/root محليًا فقط)

اتأكد الاتصال بـwp core is-installed وwp core version (رجّعت 7.0) قبل أي تعديل.

3. تثبيت GeneratePress وGenerateBlocks

  • wp theme install generatepress — v3.6.1 من downloads.wordpress.org، third-party، مش متتبّع بـGit.
  • wp plugin install generateblocks + activate — v2.3.0، third-party، مش متتبّعة بـGit.

4. بناء ثيم shemo-child (الوحيد المتتبّع بـGit)

اتعمل جوه الريبو نفسه (themes/shemo-child/) مطابقةً لقاعدة §11:

الملفالمحتوى

اتعمل directory junction (مش symlink — ما بيحتجش صلاحيات Administrator) من فولدر موقع LocalWP لفولدر الثيم جوه الريبو، بحيث الريبو هو مصدر الحقيقة الوحيد للكود والتعديل بيظهر فورًا على الموقع.

5. .gitignore جديد للريبو

اتضاف لأول مرة، يستثني themes/* وplugins/* بالكامل إلا themes/shemo-child/ وplugins/shemo-core/ (لمحطة لاحقة)، بالإضافة لـwp-config.php، .env، *.log، /wp/، /wp-content/uploads/.

6. التفعيل والتحقق

  • wp theme activate shemo-child → نجح، generatepress بقى parent، shemo-child بقى active
  • wp plugin list أكّد generateblocks نشط
  • فحص HTTP حي رجّع 200 OK بدون أي PHP fatal error، مع تاجلاين وrobots meta من محطة 1 لسه شغّالين، و<link> لستايل generatepress وshemo-child معًا، وbody class فيها wp-theme-generatepress wp-child-theme-shemo-child

7. حدود المحطة (ما لم يتم لمسه عمدًا)

  • مفيش تصميم بصري حقيقي بعد — الثيم الفرعي حاليًا هيكل فاضي لحد قرارات الهوية البصرية
  • مفيش أي إضافة تانية اتثبّتت (Safe SVG، Fluent Forms، SureCart، Rank Math، Wordfence، ShortPixel، Duplicator)
  • shemo-core plugin لسه منتظر محطة لاحقة — وهيّ نفس المحطة اللي لازم يتعرض فيها قرار ACF Pro من جديد
  • GeneratePress وGenerateBlocks لسه على الإعدادات الافتراضية تمامًا
  • إيميل الأدمن placeholder لسه متلموسش
8. بوابة المراجعة: المحطة 3 خلصت: اتسجّل تراجع المستخدم الصريح عن Bricks Builder المدفوع واتعتمد بديل مجاني (GeneratePress + GenerateBlocks) قبل أي تنفيذ، ثيم shemo-child اتبنى جوه الريبو وبقى الثيم النشط عبر directory junction، و.gitignore اتضاف لأول مرة. اتحقق كل حاجة بفحص HTTP حي للموقع. القرار المفتوح: ACF Pro (قرار 5) لازم يتعرض من جديد على المستخدم في بداية محطة نمذجة المحتوى. في انتظار موافقة المستخدم للانتقال إلى المحطة 4.

السياق

المحطة 5 — تثبيت الإضافات الأساسية المتبقية (Core Plugins Installation)

التاريخ: 2026-07-01 · ✅ مكتملة · النوع: تعديل فعلي على موقع LocalWP — تثبيت وتفعيل 7 إضافات third-party، بدون قرارات جديدة

السياق

كل الأدوات المطلوبة في هذه المحطة كانت معتمدة بالفعل من بوابة المحطة 2 (وبعضها مُعدَّل لاحقًا في المحطة 3/4 حسب مبدأ "تفضيل المجاني"): Safe SVG، Fluent Forms (مجاني)، SureCart (Launch مجاني)، Rank Math، Wordfence Free + 2FA، ShortPixel (مجاني)، Duplicator. لا قرار جديد كان مطلوبًا — المهمة تنفيذ وتفعيل وتحقق فقط.

1. تجهيز بيئة wp-cli

نفس الإعداد الموثّق من المحطات السابقة: wp.bat من مجلد Local الداخلي، مع PHPRC/MYSQL_HOME تشير لمجلد run\3PbB5bi_t\conf\ (run-hash اتغيّر عن المحطات السابقة — اتكتشف من جديد)، وPATH مضاف له مسارات PHP 8.3.29 وMySQL 8.4.0 وwp-cli. تم التأكد بـwp core version (رجّع 7.0) قبل أي تنفيذ.

3. التحقق من عدم وجود تعارض (end-to-end)

  • wp plugin list: كل الـ10 إضافات (7 الجديدة + generateblocks/meta-box/shemo-core من المحطات السابقة) ظاهرة active بدون أي علامة خطأ
  • الصفحة الرئيسية / → 200 OK، العنوان وtagline زي ما هما من المحطة 1، robots noindex لسه موجود (مفيش تعارض من Rank Math)
  • أرشيف /work/ (من shemo-core، المحطة 4) → 200 OK لسه شغّال
  • /wp-admin/ → 302 تحويل طبيعي لتسجيل الدخول (مفيش حجب من Wordfence Firewall)
  • /wp-login.php → 200 OK بعنوان صحيح
  • /shop/ و/checkout/ (صفحات أنشأتها SureCart تلقائيًا عند التفعيل) → 200 OK لكل منهما
  • wp rewrite flush نجح بدون أخطاء بعد كل التفعيلات، ومفيش debug.log جديد أو أخطاء PHP

4. فجوة فنية موثقة (حدود بيئة LocalWP، مش تعارض إضافات)

خرائط الموقع (sitemaps) — سواء sitemap الأصلي لووردبريس (/wp-sitemap.xml) أو sitemap Rank Math (/sitemap_index.xml) — بترجع 404. السبب: قاعدة location ~* \.(?:jpg|jpeg|gif|png|ico|xml)$ في nginx/site.conf المُولَّد تلقائيًا من LocalWP بتعامل أي رابط بامتداد .xml كملف ثابت **قبل** ما يوصل لقاعدة تحويل الـ404 لووردبريس. هذا مش خطأ من Rank Math ولا تعارض بين الإضافات — نفس السلوك بيحصل مع sitemap الأصلي لووردبريس برضه (تم التأكد). الحل بيحتاج تعديل يدوي على ملف Nginx config نفسه — تعديل بنية خادم مش إضافات، فاتأجل عمدًا بدل ما يتلمس config مُولَّد تلقائيًا بدون موافقة صريحة.

تم أيضًا وضع rank_math_wizard_completed = yes يدويًا (إجراء تنظيمي بس، مش قرار استراتيجي) لتفادي تكرار توجيه المستخدم لمعالج الإعداد التفاعلي.

5. خطوات تالية تحتاج تدخل المستخدم مباشرة

  • SureCart: يحتاج ربط/إنشاء حساب فعلي (OAuth) لتفعيل باقة Launch المجانية فعليًا
  • ShortPixel: يحتاج API key مجاني عبر تسجيل بإيميل حقيقي على shortpixel.com
  • Rank Math: معالج الإعداد التفاعلي (ربط Google Search Console/Analytics) لسه محتاج إكمال يدوي
  • Wordfence: API key اختياري (premium feed) — الوظائف الأساسية شغّالة بدونه. تفعيل 2FA فعليًا لأي مستخدم بيحتاج مسح QR code شخصي
  • إيميل الأدمن (من المحطة 1): لسه dev-email@wpengine.local placeholder — لسه مفتوح
  • صفحة Privacy Policy: لسه draft (تم التأكد) — لسه يحتاج قرار: تُكتب دلوقتي أو تتأجل لمحطة محتوى/قانونية لاحقة؟

6. حدود المحطة (ما لم يتم لمسه عمدًا)

  • مفيش تخصيص أو إعداد تفصيلي لأي إضافة بعد التفعيل الأساسي (غير rank_math_wizard_completed)
  • مفيش محتوى حقيقي اتضاف لأي صفحة (صفحات Shop/Checkout/Dashboard اتولدت تلقائيًا من SureCart نفسه)
  • إضافات لسه برّه نطاق كل المحطات لحد الآن: Cloudflare Turnstile، Complianz، GA4/GTM، أدوات CRM/قوائم بريدية
  • إيميل الأدمن placeholder وصفحة Privacy Policy draft — لسه مفتوحين بانتظار قرار/بيانات من المستخدم
7. بوابة المراجعة: المحطة 5 خلصت: كل الإضافات السبع المعتمدة من بوابة المحطة 2 (القرارات 6–12) اتثبتت وتفعّلت بنجاح بدون أي قرار جديد مطلوب — كلها بأعلى نسخة مجانية متاحة. اتحقق من عدم وجود تعارض بين الإضافات العشرة عبر فحوصات HTTP حية — كلها بدون أخطاء PHP أو تعارض. فجوة فنية واحدة اتوثقت بدقة (sitemaps بترجع 404 بسبب قاعدة Nginx ثابتة في إعداد LocalWP، مش بسبب الإضافات) وأُجِّلت بدل ما تتحل بتعديل غير موافق عليه. الفجوتين القديمتين (إيميل الأدمن placeholder، صفحة Privacy Policy draft) اتراجعوا وتأكد إنهم لسه مفتوحين، محتاجين قرار/بيانات حقيقية من المستخدم. في انتظار موافقة المستخدم للانتقال إلى المحطة 6.
