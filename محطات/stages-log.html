المحطة 8 — إعداد Complianz Consent Wizard (بانر موافقة الكوكيز فعليًا)

1. لماذا مش عبر واجهة wp-admin أو تعديل DB مباشر

Complianz 7.5.0 مالوش أوامر wp-cli جاهزة، وبيخزّن إعداداته في option واحد (cmplz_options). بدل تعديل قاعدة البيانات مباشرة (خطر على أداة قانونية حساسة)، تم فحص كود الإضافة ولقينا دالة عامة مخصصة بالظبط لهذا الغرض: cmplz_update_option( $name, $value ) — بتتحقق من صحة الحقل، بتنضّف القيمة، وبتُطلق نفس الـ hooks اللي بتتطلق لو حد ملى الفورم يدويًا. كمان cmplz_admin_logged_in() بتتحقق صراحة defined('WP_CLI') && WP_CLI كـbypass رسمي من مطوّري الإضافة نفسهم.

2. القرارات المطلوبة من المستخدم قبل التنفيذ

  • النطاق الجغرافي/القانوني: "عالمي (GDPR+CCPA+أقوى حماية)". قيد فني اتوضّح: النسخة المجانية بتسمح بنظام قانوني افتراضي واحد بس لكل الزوار (لا يوجد استهداف ديناميكي حسب الدولة — Premium فقط). القرار الفني: eu (GDPR) بوصفه أشد نظام (opt-in افتراضي)، بيغطي عمليًا متطلبات CCPA (opt-out) تلقائيًا.
  • فئات الكوكيز: "أساسي + إحصائيات + تسويق/إعلانات" — الثلاثة اتفعّلوا في البانر.
  • لغة البانر: "ثنائي (AR+EN)". فجوة فنية اتكشفت: الموقع مالوش Polylang/WPML مثبّت أصلاً (get_locale() = en_US) — تفعيل اللغة الثنائية قرار مفتوح من المحطة 2/6. البانر دلوقتي إنجليزي بس، وهيبقى ثنائي تلقائيًا لما محطة Polylang تحصل مستقبلاً.

معلومات حقيقية لازمة للمستندات القانونية (مش مفاتيح خارجية): اسم المؤسسة "Shemo Studio"، العنوان "القاهرة، مصر" فقط (بدون شارع تفصيلي). وبخصوص صفحة Privacy Policy القديمة (ID 3، draft من المحطة 6): Complianz بيربط بصفحات publish بس تلقائيًا — تعارض مع قرار "تسيبها draft". المستخدم اختار: خلي Complianz يولّد صفحة جديدة منفصلة.

3. التنفيذ الفعلي

المحطة 11 — Local Analytics Baseline via Burst Statistics Free

السياق

نطاق المحطة الأصلي كان GA4 عبر Site Kit، لكن الموقع لسه LocalWP فقط (shemostudio.local) وGoogle Site Kit محتاج تحقق ملكية عن بُعد. المستخدم اعتمد بديلًا صريحًا لهذه المحطة: Local Analytics Baseline via Burst Statistics Free، بدون Tunnel، بدون تغيير WordPress Address أو Site Address، وبدون أي حساب/اشتراك خارجي.

  • اتضاف القرار 19: Production GA4 + Search Console via Site Kit كبند إنتاج مستقل لاحق بعد الدومين والاستضافة الحقيقيين.
  • القرار 15 اتحدّث ليعكس إن Site Kit مثبتة لكن مُعطّلة مؤقتًا من المحطة 11، ولم يتم OAuth أو إنشاء GA4 Property.

2. تركيب Burst Statistics من المصدر الرسمي فقط

تم تثبيت وتفعيل burst-statistics v3.5.1 عبر wp-cli من مستودع WordPress الرسمي:

3. ضبط Cookieless Tracking

فحص كود الإضافة أكد إن الإعداد الرسمي اسمه enable_cookieless_tracking، وإن الإضافة توفر أمر wp-cli رسمي:

تم استخدام الأمر الرسمي بدل تعديل wp_options يدويًا. القراءة الرجعية أكدت:

{"enable_cookieless_tracking":1,"enable_turbo_mode":1}

  • wp_burst_devices / wp_burst_browsers / wp_burst_platforms

القيم الأساسية اتخزنت في wp_options مثل burst_options_settings وburst-current-version.

بدون تغيير عناوين WordPress وبدون Tunnel، تم استخدام proxy محلي مؤقت داخل جلسة الاختبار فقط يمرر الطلبات إلى 127.0.0.1:10005 مع Host: shemostudio.local. اتقفل بعد الاختبار، ولم يعرّض الموقع للإنترنت.

Complianz بقيت نشطة، وسكريبت البانر ما زال موجودًا في HTML. كود تكامل Burst مع Complianz يترك Burst بدون حظر عند تفعيل Cookieless Tracking:

  • استعلام SQL أولي استخدم عمودًا غير موجود، ثم اتراجع واتقرأ الـschema الحقيقي.
  • محاولة auth-cookie للوحة الإدارة فشلت داخل eval، ولم يتم إنشاء مستخدم مؤقت أو تخزين credentials.

10. حدود المحطة (ما لم يتم لمسه عمدًا)

  • لم يتم تغيير WordPress Address أو Site Address.
  • لم يتم استخدام Tunnel أو تعريض LocalWP للإنترنت.
  • لم يتم إنشاء GA4 Property أو تنفيذ OAuth أو ربط حساب Google.

Site Kit اتعطل مؤقتًا لإزالة أي تلميح/تحميل متعلق بـGoogle Analytics أو Tag Manager، مع تسجيل بند إنتاج مستقل لاحق لـGA4 + Search Console. تحقق فعليًا إن pageviews من الواجهة بتتسجل في جداول WordPress المحلية (wp_burst_statistics, wp_burst_sessions) وإن Complianz ما اتعارضش مع التتبع cookieless. لم يتم تغيير URLs، ولم يُستخدم Tunnel، ولم يتم إنشاء GA4 Property. في انتظار جلسة جديدة للمحطة 12 فقط، بدون تنفيذ أي محطة إضافية هنا.
